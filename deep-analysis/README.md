# 深度分析 — 对话式系统分析技能

> **版本**: 2.0 | **更新**: 2026-03-14 | **作者**: AG

## 解决的问题

在分析复杂系统时，AG 需要一种方法来深入理解问题空间并产出可行动的分析结论。但 v1 的 Analyst+Challenger 对抗模型在实际使用中失败了：

- **对抗产出共识废话** — 两个 agent 像不关心结果的辩论家，AG 综合时选"两边都没反对的安全项"，有价值但有风险的选项被系统性消除
- **AG 代替用户避险** — 有风险的决策被标为"后续再说"（= 永远不做），而不是呈现给用户让用户判断
- **框架重写用户意图** — AG 用 6-Lens 生成 aspects，本质是在 rewrite 用户需求而非 understand

6 次使用中仅 1 次真正成功交付功能（17%），核心瓶颈不在分析质量，而在分析产出是否能让下游使用。

## 解决方案

3 步流程，核心哲学是 **"对话式理解意图 + 产出带 tradeoff 的选项 + 用户做关键决策"**：

```
Intent Elicitation → Focused Investigation → Decision Surface
```

1. **Intent Elicitation**: AG 和用户多轮对话，搞清楚真实需求（不套框架）
2. **Focused Investigation**: 委派调查员深入调查用户关心的具体问题
3. **Decision Surface**: 产出带决策点和 tradeoff 的 system_map，用户逐一判断

## 技能设计哲学

### 核心信念

> **分析的目标是产出最好的方案并把风险提前暴露让用户确认，而不是产出一份安全但无用的共识文档。**

| 原则 | 含义 |
|------|------|
| **意图优先** | 先通过对话搞清用户想解决什么，再决定分析什么 |
| **选项 + Tradeoff** | 每个发现给选项和代价，不是给一个"正确答案" |
| **风险 = 决策信号** | 风险是用户需要知道的信息，不是 AG 替用户回避的东西 |
| **AG 不替用户做决策** | AG 梳理决策点和选项，用户拍板 |
| **下游可用** | system_map 必须让 system-design 或执行者直接使用 |

### 填补了什么空白？

AG 单独分析时有两个倾向：(1) 直接给"最显然"的结论，跳过其他选项；(2) 遇到风险就保守回避。这个技能通过 subagent 深入调查 + 决策面结构化呈现来对抗这些倾向。

### 什么时候可以退役？

如果 AG 底层模型能原生做到：(1) 多轮对话自然追问意图、(2) 自动为每个建议列出替代方案和 tradeoff、(3) 不自行避险而是呈现选项——那么这个技能可以简化为提示词。

### 退役后什么应该保留？

- **决策面格式**（Decision Surface）：决策点 + 选项 + tradeoff + 用户判断
- **意图对话模式**：先追问、后分析，不跳过理解
- **调查员 prompt 规范**：每个发现必须带选项而非结论

## 设计决策

### 为什么从 Analyst+Challenger 改为单一 Investigator？

**v1 做法**: Analyst 写报告 → Challenger 挑毛病 → AG 取交集。

**为什么失败**: 对抗模型的激励结构是错的——两个 agent 都不为最终结果负责。Analyst 写完就走，Challenger 只管挑毛病。AG 综合时选"两边都没反对的部分"，系统性地丢弃所有大胆但有风险的方案。

在 `692b1f28`（strategy_zoo 清理）中，Challenger "纠正"了 Analyst 4 个假设后，核心改造（artifacts/ 新建、build 合并）全部被标为"后续再说"，只做了删垃圾文件的零风险操作。

**v2 做法**: 单一调查员，任务是"回答用户的具体问题，每个回答带选项和 tradeoff"。风险不再是对抗中的"子弹"，而是需要用户决策的信号。

### 为什么去掉 6-Lens 框架？

**v1 做法**: AG 用 6-Lens（数据流/控制流/人机触点/故障域/外部依赖/非功能约束）生成 aspects。

**为什么失败**: 用户说"帮我分析一下"，AG 立刻套框架产出 8 个 aspects——但这些 aspects 是 AG 认为重要的，不是用户关心的。在 `a15109cc` 中，用户想看"哪些设计面向 agent-native"，AG 却产出了覆盖所有维度的 10 条 aspects，用户不得不手动修正。

**v2 做法**: 先对话搞清意图，分析维度自然从对话中浮现。如果某个 6-Lens 维度确实重要，AG 会在对话中提出——但不是机械套用。

### 为什么 Intent Elicitation 要多轮对话？

之前的 "USER CHECKPOINT" 是展示 AG 已经做好的东西让用户说 yes/no。但用户说 yes 不代表 AG 理解对了——用户可能只是因为 AG 产出看起来"合理"就同意了。

多轮追问的价值：AG 问"你最不满意什么？"，用户可能提到一个 AG 完全没想到的痛点。这种信息在"展示 aspects 让用户确认"的模式中永远不会浮现。

## 失败的尝试

### v1: Analyst + Challenger 对抗模式

已在"设计决策"部分详述。核心教训：**对抗式分析的激励结构错误，导致有价值的高风险选项被系统性消除。**

### v1: 6-Lens 框架驱动的 Aspect Discovery

已在"设计决策"部分详述。核心教训：**套框架 ≠ 理解用户意图，AG 应该先追问再分析。**

### SPIKE 自验模式（system-design 侧，但影响 deep-analysis 产出）

`a15109cc` 中 deep-analysis + system-design 产出了 `agent-native-product` skill，SPIKE 验证"通过"。但后续 3 个对话中用户实际使用发现 skill 完全不可用（checklist 方法论不对、产出太粗糙）。核心教训：**自己验自己不算验证。**

## FAQ

**Q: 调查员可以用多个吗？**

可以，当问题域足够大时可以分派 2 个调查员分别负责不同子域。但它们是协作关系（各管各的领域），不是对抗关系（互相挑毛病）。

**Q: 和 system-design 怎么衔接？**

deep-analysis 的 system_map（含用户已做的决策）直接作为 system-design 的输入。因为用户已经在 Decision Surface 环节做了关键判断，system-design 拿到的输入是有明确方向的，不是一堆"安全共识"。

**Q: 如果用户的意图就是"帮我看看"这种模糊需求怎么办？**

AG 的工作就是帮用户把模糊变具体。追问："看什么？你最近遇到了什么问题？你希望分析完之后做什么？"。几轮对话后通常能抵达具体的问题陈述。

**Q: 对于纯研究性质的分析（如 Dota 2 AI），Decision Surface 格式合适吗？**

合适。即使是研究性分析，也应该产出"方案 A/B/C 各有什么 tradeoff"而不是"推荐方案 X"。用户决定是否以及如何行动。
